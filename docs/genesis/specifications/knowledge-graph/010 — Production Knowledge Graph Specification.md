Genesis Foundational Standards (GFS)
GFS-010 — Production Knowledge Graph Specification

Document ID: GFS-010
Title: Production Knowledge Graph Specification
Version: 1.0.0
Status: Foundational Standard
Authority: Derived from GFS-000 through GFS-009

1. Purpose

This Specification defines the structure, semantics, serialization format, and validation requirements for the Production Knowledge Graph (PKG).

The PKG is the canonical representation of all production intelligence within Genesis. Every document, specification, and decision generated by Genesis is a materialized view of the PKG.

2. Foundational Principle

The PKG is the single source of truth.

No production decision exists outside the PKG. Every downstream system derives its authority from the PKG. If a decision exists only in a document, prompt, or conversation, it has not been formalized.

3. Graph Structure

The PKG is a directed labeled property graph composed of:

3.1 Nodes

Each node represents a concept defined in the Genesis Ontology. Every node must have:

- id: UUID (assigned by Genesis, immutable)
- type: OntologyClass (from GO-001 Core Ontology)
- label: Human-readable name
- properties: Key-value map (type-constrained per ontology)
- confidence: One of {EXPLICIT, INFERRED, CONFIRMED, ASSUMED, UNKNOWN}
- created_at: ISO 8601 timestamp
- provenance: Reference to the agent/role that created the node

3.2 Edges

Each edge represents a semantic relationship from the Genesis Semantic Relationship Catalog (GO-002). Every edge must have:

- id: UUID
- type: RelationshipType (from GO-002)
- source_id: UUID (must reference an existing node)
- target_id: UUID (must reference an existing node)
- properties: Key-value map
- confidence: One of {EXPLICIT, INFERRED, CONFIRMED, ASSUMED, UNKNOWN}
- created_at: ISO 8601 timestamp
- provenance: Reference to the agent/role that created the edge

3.3 Subgraphs

A subgraph is a named, versioned collection of nodes and edges that represents a complete production concern. Every production must contain at least the following subgraphs:

- Narrative Subgraph: Story structure, acts, sequences, scenes, beats
- Character Subgraph: All character nodes and their relationships
- World Subgraph: Environment, setting, temporal context
- Audience Subgraph: Target audience, emotional journey, experience design
- Production Subgraph: Schedule, resources, dependencies, milestones

4. Serialization Format

The PKG is serialized as a JSON-LD document conforming to the Genesis Ontology context.

4.1 Top-Level Structure

{
  "@context": "https://genesis.movieos.dev/ontology/v1/context.jsonld",
  "id": "uuid",
  "production": "production-name",
  "version": "1.0.0",
  "created_at": "ISO 8601",
  "provenance": { "agent": "agent-name", "session": "session-id" },
  "nodes": [ ... ],
  "edges": [ ... ],
  "subgraphs": { ... },
  "validation": { ... }
}

4.2 Node Serialization

{
  "id": "uuid",
  "type": "go:Scene",
  "label": "The First Crack",
  "properties": {
    "go:sceneNumber": 5,
    "go:duration": 7.0,
    "go:energy": 4,
    "go:phase": "crack"
  },
  "confidence": "CONFIRMED",
  "created_at": "2026-07-19T12:00:00Z",
  "provenance": { "agent": "StoryArchitectAgent", "session": "sess-001" }
}

4.3 Edge Serialization

{
  "id": "uuid",
  "type": "go:precedes",
  "source_id": "uuid-of-scene-4",
  "target_id": "uuid-of-scene-5",
  "properties": { "go:reason": "narrative escalation" },
  "confidence": "CONFIRMED",
  "created_at": "2026-07-19T12:00:00Z",
  "provenance": { "agent": "ScenePlannerAgent", "session": "sess-001" }
}

5. Validation Requirements

Every PKG must pass the following validations before it can be certified as production-ready:

5.1 Structural Validation

- All node IDs are valid UUIDs
- All edge source/target IDs reference existing nodes
- No dangling references
- All required properties are present per ontology class
- All property values conform to type constraints

5.2 Semantic Validation

- No contradictory relationships (e.g., a character cannot be both protagonist and antagonist)
- All temporal relationships form a consistent timeline
- All causal chains are complete (no effect without cause)
- Character arcs are internally consistent

5.3 Completeness Validation

- All required subgraphs are present
- Each subgraph meets its minimum node count
- All confidence thresholds are met (no UNKNOWN in critical paths)
- All dependencies between subgraphs are satisfied

6. Versioning

The PKG is versioned as a whole. Each version must be immutable once certified.

Version numbering: MAJOR.MINOR.PATCH
- MAJOR: Incompatible structural changes
- MINOR: Additive changes (new nodes, edges, subgraphs)
- PATCH: Confidence upgrades, corrections, refinements

7. Production Knowledge Package

A Production Knowledge Package (PKP) is the distributable unit of production intelligence. It contains:

- The PKG JSON-LD document
- A manifest file describing the package contents
- All materialized views (screenplay, storyboard, production plan, etc.)
- A validation certificate signed by the Governance Agent
- A provenance log of all decisions

8. Storage

The PKG is stored in a graph database that supports:

- Property graph queries
- Subgraph extraction
- Temporal queries (version history)
- Confidence-based filtering
- Provenance tracing

The reference implementation uses a combination of:
- SQLite for metadata and versioning
- JSON-LD files for distribution
- In-memory graph for active reasoning sessions

9. Compliance

Every Genesis agent must read from and write to the PKG. No agent may maintain its own independent knowledge store. Temporary working memory is permitted during a reasoning session but must be committed to the PKG before the session concludes.
