Genesis Ontology Registry
GO-200 — Generated Ontology Registry

Document ID: GO-200
Title: Generated Ontology Registry
Version: 1.0.0
Status: Registry
Authority: Derived from GFS-009 Constitutional Ontology Framework

1. Purpose

This registry catalogs ontologies that have been auto-generated by the Genesis Compiler from non-ontology sources: schemas, specifications, production manifests, and accumulated pattern libraries. Generated ontologies are derivative artifacts. They do not hold constitutional authority. They are registered here so that the engine, downstream tooling, and human reviewers can discover, cite, and validate them.

2. Scope

This registry covers:
- Ontologies derived from GSS-NNN schema specifications.
- Ontologies derived from GSPEC-NNN format or protocol specifications.
- Ontologies derived from production manifests (per-PKG schemas).
- Ontologies derived from validated pattern libraries (GWS-013).
- Ontologies derived from external standard mappings (GREF-002).

This registry does NOT cover hand-authored ontologies in the GO-001 through GO-119 family. Those are registered in the ontology core/domain directories directly.

3. Registry Entry Format

Each entry contains:
- Generated Ontology ID: GO-NNN assigned by the compiler.
- Source: The non-ontology document(s) the ontology was derived from.
- Generated By: Genesis Compiler version and run identifier.
- Generated At: Timestamp.
- Digest: Content digest of the generated ontology.
- Parent Ontology: The hand-authored ontology this generated ontology extends or conforms to.
- Status: active | superseded | retired.
- Superseded By: Ontology ID, if superseded.

4. Generation Rules

4.1 From Schemas (GSS-NNN)
- Each GSS schema becomes an OWL ontology with classes for each schema object and properties for each field.
- SHACL shapes are emitted alongside the OWL ontology for validation.
- The generated ontology declares its parent as the domain ontology covering the same subject.

4.2 From Specifications (GSPEC-NNN)
- Each specification that defines a data model produces a generated ontology fragment.
- Specifications that define protocols produce generated event ontologies compatible with AsyncAPI.

4.3 From Production Manifests
- Each certified PKP's manifest produces a per-production ontology fragment that records the structure of that PKP.
- These fragments are used for cross-PKG queries and for archival retrieval.
- They are retired when the PKP is withdrawn.

4.4 From Pattern Libraries
- Validated patterns in the pattern library produce a generated pattern ontology.
- The pattern ontology is regenerated whenever the pattern library version changes (GWS-013).
- The prior version is marked superseded, not deleted.

4.5 From External Standard Mappings
- Each standard mapped in GREF-002 can produce a generated bridge ontology linking Genesis terms to the external standard's terms.
- Bridge ontologies are regenerated whenever the mapping or the external standard version changes.

5. Validation of Generated Ontologies

- Every generated ontology must be syntactically valid OWL.
- Every generated ontology must conform to its parent ontology's constraints.
- Every generated ontology must pass SHACL validation against the active GSS schemas.
- Generated ontologies that fail validation are recorded with status retired and a defect report. They are not used by the engine.

6. Versioning

- A generated ontology version is the concatenation of the source document version and the compiler run identifier.
- Regenerating from an unchanged source produces the same digest and the same version.
- Regenerating from a changed source produces a new version; the prior version is superseded.
- Superseded versions remain queryable for historical PKPs but are not used for new work.

7. Lifecycle

- active: The ontology is current and used by the engine.
- superseded: A newer version exists. The ontology is retained for historical queries.
- retired: The ontology failed validation or its source was withdrawn. It is retained only for audit.

8. Registry Access

- The registry is queryable by agents and tooling.
- Queries support filtering by source, parent ontology, status, and digest.
- The registry is part of the engine deployment record and is included in audit exports.

9. Initial Registry

At v1.0.0, the registry contains no active entries. Entries are added by the Genesis Compiler as it generates ontologies during deployment (GWS-011) and learning cycles (GWS-013). The first deployments will typically produce entries derived from the active GSS schema set and the GREF-002 standards mapping.
